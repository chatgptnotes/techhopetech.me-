DevOps Glossary

Plain-English definitions for the Scrum / Agile vocabulary used across Boards, Backlog, Sprints, Reports, Hierarchy, Calendar, and Retro.

Work-item hierarchy parent → child

Items nest top-down. Each level should fit inside its parent. See it visually on the Hierarchy page.

Epic

A large body of work that delivers a strategic outcome. Spans multiple sprints. Broken into Features and Stories.

e.g. "Replace the entire BNI member onboarding flow."

Feature

A deliverable chunk inside an Epic. Usually fits in one or two sprints. Made of Stories.

e.g. "Self-serve member sign-up form with KYC upload."

Story (user story)

A user-facing slice that delivers a small visible value end-to-end. Fits in one sprint. Format: "As a <role>, I want <action>, so that <benefit>."

e.g. "As a member, I want to upload my Aadhaar so that my profile is KYC-verified."

Task

Developer-facing work to implement part of a Story. No direct user value on its own. Fits in a day or two.

e.g. "Add /api/kyc-upload endpoint with mime-type validation."

Sub-task

Even smaller breakdown of a Task, usually used when two devs are pairing or splitting work.

e.g. "Write the file-size limit unit test."

Bug

Something that's broken in shipped or in-progress work. Can sit at any level. Has its own state machine: Reported → Triaged → In Progress → Verified → Closed.

e.g. "PDF export fails for files larger than 10 MB."

PBI (Product Backlog Item)

Generic Scrum term for anything in the Product Backlog. In our tool, every Epic / Feature / Story / Task / Bug is a PBI when it's not yet pulled into a sprint.

"PBI" and "Backlog item" mean the same thing here.

Backlog

The ordered list of everything we plan to do but haven't pulled into the active sprint yet. Highest priority at the top. Lives at Backlog.

Workflow states columns on the Board

Every work item moves through these states left to right. Visualised on Boards.

To Do

In the sprint but not yet started.

In Progress

A developer is actively working on it. WIP limits per developer apply (Scrum default: one Story per dev at a time).

Review

Code is up for review (PR open) or QA is testing. Not done until reviewer signs off.

Done

Meets the team's Definition of Done: merged, deployed, tested, no known regressions. Counts toward sprint velocity.

Definition of Done (DoD): the team's agreed checklist that an item must satisfy before it can move to Done. Typically: code reviewed, tests passing, deployed to staging, accessibility checked, docs updated.

Sprints & planning time-boxed iterations

Sprint

A fixed-length iteration (typically 1 or 2 weeks) where the team commits to a set of items and aims to ship them. Has a start date, end date, and a goal. State: planned → active → completed.

Sprint goal

One sentence describing the outcome of this sprint. Helps the team say "no" to scope creep mid-sprint.

e.g. "Members can sign up self-serve and upload their KYC documents."

Capacity

How many story points each developer can realistically commit to this sprint, accounting for leaves, holidays, and meetings. Visible in Sprints.

Story points

A relative-effort estimate (1, 2, 3, 5, 8, 13 …). Not hours. Forces the team to compare items rather than guess clock time.

Velocity

Average story points completed per sprint over the last few sprints. Used to forecast how much fits in the next one.

Resource leave

Time off for a developer (vacation, sick, public holiday, training). Reduces effective capacity for the affected sprint. Manage in Calendar.

Scrum ceremonies recurring meetings

Sprint Planning

Start of the sprint. Team picks items from the top of the Backlog up to capacity, agrees the Sprint Goal.

Daily Standup

15-minute sync. Each dev answers: what did I do yesterday, what will I do today, what's blocking me?

Sprint Review

End of the sprint. Demo what's Done to stakeholders. Collect feedback for the Backlog.

Retrospective (Retro)

After the Review. Team reflects on the sprint itself: what to keep, what to change. Captured in Retro using the Four L's.

The Four L's

A retrospective format with four columns: Liked (what worked), Learned (what we figured out), Lacked (what was missing), Longed for (what we wish we'd had). Each team member adds notes; team groups themes; agrees one or two actions for next sprint.

Backlog Refinement (Grooming)

Mid-sprint hour where the team estimates and clarifies upcoming Backlog items so the next planning runs fast.

Metrics & charts how we measure

Burndown

Line chart of remaining work versus days in the sprint. Should trend down to zero by sprint end. Visible in Sprints.

Burnup

Mirror of burndown. Plots completed work climbing toward total scope. Better at showing scope changes mid-sprint.

Cycle time

How long it took an item from "In Progress" to "Done." Lower is better, with caveats.

Lead time

From "created" to "Done." Shows how responsive the whole pipeline is, not just dev work.

Throughput

Items completed per unit time (per week, per sprint). Useful when items are similarly sized.

WIP (Work In Progress)

How many items are currently In Progress. Should be small. Per-developer WIP limits prevent thrash.

This vocabulary is Scrum-flavoured. Other agile flavours (Kanban, XP, SAFe) use overlapping but slightly different terms. When in doubt, use the simplest term that fits the work, not the most impressive-sounding one.
